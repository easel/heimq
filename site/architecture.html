Architecture

One distribution, one embeddable engine, one evidence trail.

heimq separates the memory-first CLI distribution from the engine crates that downstream systems embed. Release packaging and docs stay close to the same commands and workflows readers can inspect.

Distribution The heimq CLI runs a single-node, memory-first broker that accepts Kafka-compatible traffic on port 9092.
Engine crates heimq-wire and heimq-broker provide wire scaffolding and broker semantics over pluggable backend traits.
Conformance heimq-testkit and parity workloads turn compatibility into repeatable evidence.
Binary distribution Release workflow packages the Linux x86_64 artifact and publishes it through GitHub Releases.
Container distribution Docker workflow tags and publishes the image to GHCR after a smoke test passes.
Chart distribution Helm workflow packages the chart and pushes it to the GHCR OCI registry.
Repo layers

What lives where

  • crates/ holds the Rust implementation and test crates.
  • crates/heimq/docs/helix holds the product vision, PRD, contracts, tests, and design context.
  • charts/heimq holds the deployment chart.
  • .github/workflows holds release and validation automation.
  • site/ holds the static documentation pages.

Design principle

Keep public claims close to evidence. The site mirrors README commands, links to HELIX product context, and points compatibility statements at tests and contracts.

That makes the site useful for first-time readers while still allowing the README to remain the authoritative source for copy-paste commands.
