    <title>Async produce path: stop blocking a worker thread per in-flight append</title>
    <description>
PROBLEM: LogBackend::append / PartitionLog::append are synchronous, so a Produce request blocks its broker worker thread until the backend returns. For backends that DEFER the durable commit — e.g. fjord&#39;s CoordinatorLogBackend, whose server-side flush buffer (group commit) holds the append until a batched commit_object completes — the number of in-flight appends, and therefore the achievable group-commit / object-coalescing degree, is bounded by the broker&#39;s worker-thread count rather than by client concurrency.&#xA;&#xA;EVIDENCE (fjord): with client batching disabled (linger.ms=0, batch.num.messages=1), 32 concurrent producers coalesced only ~15x with 16 worker threads — the degree tracked worker count, not the 32 connections. Sync append caps the win of deferred-commit backends.&#xA;&#xA;DESIGN: add an async append entry point on the storage seam (e.g. `async fn append_async(..) -&gt; Future&lt;(offset,count)&gt;` that the produce router awaits, or offload the sync append onto a bounded spawn_blocking pool). The backend signals completion (offset assigned) via a Future/oneshot instead of blocking the calling worker. Keep the existing sync `append` working for simple backends via a default impl that bridges to the async one (or vice-versa). fjord&#39;s Flusher already uses a background flush thread + a per-append channel — it would hand back the receiver&#39;s Future directly instead of blocking on recv(). Must preserve per-(producer,partition) ordering and idempotent sequencing and not regress the in-memory/object-log backends.&#xA;&#xA;Discovered while building fjord&#39;s TD-005 server-side flush buffering (fjord commits 1c33261 / 05ee78c).
    </description>
    <acceptance>
Produce handler no longer holds a worker thread for the duration of a deferred backend commit; a single broker with N worker threads sustains many more than N concurrent in-flight appends; fjord&#39;s flush_coalescing benchmark shows coalescing degree scaling with client concurrency rather than broker worker-thread count; per-(producer,partition) ordering and idempotent sequencing preserved; all conformance/parity tests stay green; simple in-memory/log backends keep working via a default sync bridge.
