Deployment

Release artifacts land in GitHub Releases, GHCR, OCI, and Pages.

The repository uses a small number of predictable publish targets so the release story stays easy to verify and explain.

Artifacts

What publishes where

  • Binary release: .github/workflows/release-binaries.yml attaches the Linux x86_64 binary and checksum to a GitHub Release.
  • Docker image: .github/workflows/docker-image.yml publishes tagged images to ghcr.io/easel/heimq.
  • Helm chart: .github/workflows/helm-chart.yml packages and pushes oci://ghcr.io/easel/charts/heimq.
  • Docs site: .github/workflows/pages.yml publishes the checked-in site/ directory on pushes to main.

Release tags

The plan maps vX.Y.Z tags to binary, Docker, and chart versions with the same semantic version, so operators can trace a deployment back to one release point.

Stable tags also receive the floating Docker aliases X.Y, X, and latest.
Workflow

Pages publication path

The Pages workflow is intentionally narrow: checkout, confirm the site exists, upload site/ as the Pages artifact, then deploy it to the github-pages environment. 

on:
  push:
    branches:
      - main

That keeps the site published from the branch that carries the release infrastructure plan and the README links.
