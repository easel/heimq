    <title>Avoid double decoding Fetch requests on the hot path</title>
    <description>
Fetch requests currently take an avoidable double-decode path. Router::route_async decodes the request header and then handle_fetch_long_poll decodes the FetchRequest to inspect max_wait_ms and requested offsets. When data is available it calls route(data), which decodes the header again and fetch::handle decodes the FetchRequest again. This is cheap per request but visible when consume issues many fetches. Refactor the fetch route so the decoded header/body and, where possible, decoded FetchRequest metadata are reused instead of decoding the same frame multiple times. Preserve max_wait_ms long-poll behavior and existing wire compatibility.&#xA;&#xA;In-scope files:&#xA;- crates/heimq/src/protocol/router.rs&#xA;- crates/heimq/src/handler/fetch.rs&#xA;- crates/heimq/src/handler/tests.rs or existing focused tests, if needed&#xA;- crates/heimq/tests/bench_baseline.rs only if needed to keep diagnostics compiling&#xA;&#xA;Out-of-scope:&#xA;- Changing Kafka Fetch response semantics.&#xA;- Storage layout or copy-reduction work.&#xA;- Consumer group rebalance behavior.
    </description>
